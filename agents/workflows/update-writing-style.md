# 工作流：迭代写作风格

## 触发条件

用户手动修改了 Agent 生成的笔记内容，或主动要求调整写作风格偏好。

## 步骤

### 1. 收集差异

获取用户修改前后的内容。方式：

- `git diff`：如果修改已保存到文件
- 用户指定：用户直接指出修改了哪些地方
- Agent 对比：如果 Agent 有之前生成的版本，直接对比

### 2. 分析修改模式

对差异进行分类：

- 用词替换：用户删除或替换了哪些表述
- 句式调整：长句拆短、短句合并、语序变化
- 结构改动：列表层级重组、条目位置调整
- 公式排版：行内/行间公式选择、符号约定变化
- 其他：不属于以上类别的修改

### 3. 提炼规则

从具体修改中抽象出通用规则：

- 不是记录"把 X 改成 Y"，而是归纳出"偏好 A 风格而非 B 风格"
- 每条规则应有正/反示例，而非仅抽象描述
- 如果同一次修改中发现多条规则，逐一列出

将提炼结果展示给用户确认。

### 4. 更新风格文档

将用户确认的规则写入 `agents/styles/writing-style.md` 的对应章节：

- 用词替换 → 风格要点或禁止清单
- 句式调整 → 风格要点中的对应子节
- 公式排版 → 数学与文字的关系
- 新发现 → 新增子节

如果新规则与已有规则冲突，与用户讨论后决定取舍。

### 5. 精简

如果禁止清单或风格要点变得过长，归纳合并：用更精炼的概括性规则替代逐条罗列。

### 6. 拆分与重构

当 `agents/styles/writing-style.md` 膨胀到难以维护，或某些主题的规则形成独立体系时，考虑拆分为多个文件。拆分条件：

- 某一章节的内容可以独立成文，且有自己的主题（如数学排版风格、不同学科的特殊约定等）
- 单个文件过长，不利于 Agent 在每次写作前快速查阅

拆分后需同步更新 `AGENTS.md` 索引和本工作流中的文件引用。

### 7. 验证

用更新后的风格重写一小段内容（可以是刚才修改的那段），让用户确认是否对味。
